What is XSS

˙ Essentially being able to inject malicious code into a target application ˙ Injected by user input via comments, URLs, forms anywhere a user can generate content ˙ Attacks are usually trying to gain an JavaScript execution context in a user's browser ˙ Two different types stored and reflective ˙ There is an abundance of attack vectors out there

How do we mitigate against?

Deny all : Never insert user data into these locations! 


	<script> ...NEVER PUT UNTRUSTED DATA HERE...</script> 	Directly in a script

	<!--...NEVER PUT UNTRUSTED DATA HERE...--> 						Inside an HTML comment

	<div ...NEVER PUT UNTRUSTED DATA HERE...=test /> 			In an attribute name

	<NEVER PUT UNTRUSTED DATA HERE... href="/test" /> 			In a tag name

	<style>...NEVER PUT UNTRUSTED DATA HERE...</style> 		Directly in CSS


More details from OWASP

Always HTML escape before inserting user data into HTML elements! 


	& --> &amp;
	< --> &lt;
	> --> &gt;
	" --> &quot;
	' --> &#x27;     &apos; not recommended because its not in the HTML spec
	/ --> &#x2F;     Forward slash is included as it helps end an HTML entity


More details from OWASP

JavaScript escape before inserting untrusted data into variables 


	<script>alert('...ESCAPE UNTRUSTED DATA BEFORE PUTTING HERE...')</script>     Inside a quoted string
	
	<script>x='...ESCAPE UNTRUSTED DATA BEFORE PUTTING HERE...'</script>          One side of a quoted expression
	
	<div onmouseover="x='...ESCAPE UNTRUSTED DATA BEFORE PUTTING HERE...'"</div>  Inside quoted event handler

Cross Origin Resource Sharing

I think we all know what CORS is, but what are the problems we need to be aware of? ˙ Universal Allow: ( * Wildcard , vulnerable intranet , 3rd party attacks ) ˙ Site-level Cross Origin Access: ( Ensure only the pages that need COR have the header ) ˙ Access-control decision based on Origin header: ( Do not rely on origin header to be true eg. COR     from website 'origin.com' gets data - use authentication ) ˙ Prolonged caching of Preflight responses: ( eg. If access control header is changed on server the     browser will follow the pervious cached one. 30 mins max ) ˙ Misplaced-trust ( If a trusted site is compromised they can potentially use that trust to attack you ) ˙ Processing rogue COR ( If not COR supported make sure to reject these requests - otherwise     application DDOS can occur )

Cookie - Bad

Problemz? ˙ This data is completely in control of the client- it is trivial to change ˙ By default are not encrypted and so over HTTP can be snooped

Cookie - Good

Solutions? ˙ Don't store sensitive information in cookies ˙ Restrict cookie to only your application / application paths or the sub domains that need it ( using    domain and path ) ˙ Prevent XSS by using HttpOnly flag this lets newer browsers restrict access to cookies by JavaScript ˙ Use the Secure Connection options so they can't accidently be sent via HTTP and cleartext
More details from Treehouse

Iframes

˙ Prevent how your sensitive pages are display with X-Frame-Options http response header ( ALLOW ,     DENY , SAMEORIGIN and ALLOW-FROM ) ˙ Use the principle of least privilege with the sandbox attribute ( sandbox="allow-same-origin
    allow-scripts allow-popups allow-forms" )
